[2] ⚠️ BREAKING — เกณฑ์ความเสี่ยงเปลี่ยน
================================================================================

เดิมเฟิร์มแวร์ใช้:   LOW <= 11 วิ | MODERATE <= 20 วิ | HIGH > 20 วิ
เวอร์ชันใหม่ใช้:     LOW <= 11 วิ | MODERATE <= 30 วิ | HIGH > 30 วิ

เหตุผล: เอกสารโครงการหัวข้อ 6.5.1 ระบุเกณฑ์ที่ "โรงพยาบาลต้องการ" ไว้ว่า
    เวลาที่ได้ <= 11 วินาที      = ไม่มีความเสี่ยงต่อการหกล้ม
    เวลาที่ได้ > 11 - 30 วินาที  = มีความเสี่ยงต่อการหกล้ม
    เวลาที่ได้ > 30 วินาที       = มีความเสี่ยงสูงมากต่อการหกล้ม
ค่า 20 ในโค้ดเดิมไม่ตรงเอกสาร ทำให้คนที่ทำได้ 25 วินาทีถูกจัดเป็น HIGH
ทั้งที่ตามเกณฑ์ต้องเป็น MODERATE

สิ่งที่เว็บต้องทำ:
  • ค้นหาเลข 20 ที่ใช้เป็น threshold ในโค้ดเว็บ แล้วเปลี่ยนเป็น 30
  • ทางที่ดีกว่าคือ "อย่าคำนวณเอง" — ใช้ field risk_level ที่บอร์ดส่งมาโดยตรง
    จะได้มีแหล่งความจริงแหล่งเดียว ไม่มีทางเพี้ยนกัน
  • ถ้าจำเป็นต้องคำนวณเองบนเว็บ (เช่น ตอนเฉลี่ย 3 รอบ) ให้ประกาศเป็นค่าคงที่
    ตัวเดียวในโปรเจกต์ อย่ากระจายไปหลายไฟล์

⚠️ ข้อมูลเก่าที่บันทึกไว้ก่อนหน้านี้ใช้เกณฑ์ 20 อยู่ ถ้าจะเทียบย้อนหลังต้อง
   คำนวณ risk_level ใหม่จาก total_sec ไม่ใช่เชื่อค่าที่เก็บไว้

[3] Collection: tug_results  (บอร์ดเขียน / เว็บอ่านอย่างเดียว)
================================================================================

หนึ่ง document = การทดสอบหนึ่งรอบ (หนึ่ง trial) ไม่ใช่หนึ่ง session

document ID: "<finished_at>_<trial_no>"   เช่น "1753012345_2"

  ⚠️ BREAKING: เดิมใช้ String(millis()) เป็น ID ซึ่งเป็นบั๊ก เพราะ millis()
     จะกลับไปเริ่มนับที่ 0 ทุกครั้งที่บอร์ดรีบูต (และตอนนี้เว็บสั่งรีบูตได้ด้วย)
     ทำให้การทดสอบรอบแรกหลังรีบูต "เขียนทับ" document เก่าโดยไม่มีใครรู้
     และเรียงลำดับตามเวลาไม่ได้เลย
     ตอนนี้ใช้ epoch จาก NTP แทน จึงเรียงเวลาได้และไม่ชนกัน

fields ทั้งหมด:

  ชื่อ field        ชนิด Firestore   คำอธิบาย
  ---------------- ---------------- ---------------------------------------------
  subject_key      stringValue      รหัสผู้เข้าทดสอบ (เว็บเป็นคนกำหนด, ดูข้อ 6)
                                    ค่า "unassigned" = เว็บยังไม่ได้ระบุตัวคน
  session_id       stringValue      รหัสชุดการทดสอบ (1 session = 3 trials)
  trial_no         integerValue     ลำดับรอบใน session นี้ เริ่มที่ 1
  checkpoint_sec   doubleValue      เวลาจากลุกขึ้น → ถึงจุดหมุนตัว 3 เมตร
  return_sec       doubleValue      เวลาจากจุดหมุนตัว → กลับมานั่ง  [ใหม่]
  total_sec        doubleValue      เวลารวมทั้งรอบ (= ค่า TUG time)
  risk_level       stringValue      "LOW" | "MODERATE" | "HIGH"
  status           stringValue      "completed" | "aborted"          [ใหม่]
  started_at       integerValue     epoch วินาที ตอนเริ่มจับเวลา      [ใหม่]
  finished_at      integerValue     epoch วินาที ตอนจบ               [ใหม่]
  device           stringValue      "chair"
  fw_version       stringValue      เช่น "chair-2.0.0"               [ใหม่]

  (checkpoint_sec / total_sec / risk_level มีอยู่เดิม — ที่เหลือเป็นของใหม่ทั้งหมด)

⚠️ status = "aborted" คืออะไร
   บอร์ดจะบันทึกรอบที่ล้มเหลวไว้ด้วย ไม่ทิ้งเงียบ ๆ เกิดได้ 2 กรณี:
     • หมดเวลา 120 วินาที (ผู้ทดสอบล้ม / เดินออกนอกเส้น / ไม่กลับมานั่ง)
     • เจ้าหน้าที่กด Abort บนเว็บ
   เก็บไว้เพื่อการ audit ตามที่เอกสารกำหนด แต่ **เว็บต้องกรองออกเสมอ**
   เวลาคำนวณค่าเฉลี่ยหรือแสดงผลความเสี่ยง เพราะ total_sec ของรอบ aborted
   ไม่ใช่เวลาที่ทำเสร็จจริง (เป็นแค่เวลาที่ผ่านไปก่อนโดนยกเลิก)

   → ทุก query ที่เอาไปคิดผล ต้องมี where("status","==","completed")

⚠️ กรณี started_at = 0
   เกิดเมื่อบอร์ดเริ่มทดสอบตอนที่ยังไม่ได้ sync เวลาจาก NTP (เน็ตหลุดตอนบูต)
   เว็บต้องเผื่อไว้ ให้แสดงเป็น "ไม่ทราบเวลาเริ่ม" แทนที่จะแปลงเป็น 1 ม.ค. 1970
   ซึ่งดูเหมือนบั๊ก

   ส่วน finished_at รับประกันว่า > 0 เสมอ เพราะถ้าบอร์ดยังไม่รู้เวลาจริง
   มันจะเก็บผลไว้ใน buffer ก่อน แล้วประทับเวลาตอนที่ส่งได้จริง
   (แปลว่า finished_at ของรายการที่เคยค้างจะช้ากว่าเวลาที่ทดสอบจริงเล็กน้อย
   ถ้าต้องการเวลาที่แม่นยำให้ใช้ started_at เมื่อมีค่า)

[4] Collection: device_status  (บอร์ดเขียน / เว็บอ่านอย่างเดียว)
================================================================================

มี 2 document แบบตายตัว: "chair" และ "checkpoint"

  ⚠️ ใหม่: device_status/checkpoint
     เดิมบอร์ด Checkpoint ต่อ WiFi ไว้เฉย ๆ เพื่อ sync channel ของ ESP-NOW
     ไม่ได้คุยกับ Firestore เลย เว็บจึงไม่มีทางรู้ว่ามันเปิดอยู่หรือเปล่า
     ตอนนี้มันรายงานตัวเองแล้ว เว็บควรแสดงสถานะทั้งสองบอร์ด
     (เอกสารขั้นตอนที่ 2 กำหนดให้ระบบแสดงไฟ "พร้อม/ไม่พร้อม" ของอุปกรณ์)

device_status/chair:
  online              booleanValue   บอร์ดตั้งเป็น true เสมอตอนเขียน — ดูหมายเหตุ
  last_seen           integerValue   epoch วินาที ของ heartbeat ล่าสุด
  state               stringValue    CALIBRATE|WAIT_SIT|READY|RUNNING|RETURNING|COOLDOWN
  rssi                integerValue   ความแรงสัญญาณ WiFi (dBm, ติดลบ)
  device              stringValue    "chair"
  fw_version          stringValue    "chair-2.0.0"
  uptime_sec          integerValue   เปิดมากี่วินาทีแล้ว (ใช้ดูว่าเพิ่งรีบูตไหม)
  checkpoint_online   booleanValue   Chair ยังคุยกับ Checkpoint ได้ไหม (ผ่าน ESP-NOW)
  pending_uploads     integerValue   จำนวนผลที่ค้างอยู่ในบอร์ด ยังส่งไม่สำเร็จ
  armed               booleanValue   พร้อมเริ่มจับเวลาหรือยัง (ดูข้อ 5)
  subject_key         stringValue    คนที่บอร์ดกำลังจะทดสอบ
  session_id          stringValue    session ปัจจุบัน
  trial_no            integerValue   รอบถัดไปที่จะทำ

device_status/checkpoint:
  online, last_seen, state, rssi, device, fw_version, uptime_sec
  state ของ checkpoint คือ IDLE | DETECTING | RESULT
  chair_online        booleanValue   Checkpoint ยังคุยกับ Chair ได้ไหม

⚠️ วิธีตัดสินว่า "ออนไลน์" — ห้ามเชื่อ field `online` เพียงอย่างเดียว
   field นี้ถูกเขียนโดยบอร์ดตอนที่ยังมีชีวิตอยู่ ถ้าบอร์ดไฟดับ มันจะค้างเป็น
   true ตลอดไป เพราะไม่มีใครไปเขียนให้เป็น false ได้
   ต้องเช็คจาก last_seen เทียบเวลาปัจจุบันเสมอ:

       const HEARTBEAT_SEC = 15;
       const isOnline = (Date.now()/1000 - last_seen) < HEARTBEAT_SEC * 3;  // ~45 วิ

   ที่ใช้ 3 เท่า เพราะเผื่อ heartbeat พลาดไป 1-2 ครั้งจากเน็ตกระตุก
   จะได้ไม่ขึ้น OFFLINE หลอก ๆ ให้เจ้าหน้าที่ตกใจ

⚠️ Checkpoint อาจขึ้น OFFLINE ชั่วคราวระหว่างการทดสอบ — ไม่ใช่บั๊ก
   ตอนที่ Checkpoint อยู่ในสถานะ DETECTING (เฝ้ารอผู้ทดสอบเดินมาถึง) มันจะ
   หยุดส่ง heartbeat ชั่วคราว เพราะการเรียก HTTPS จะบล็อกลูปหลายร้อย ms
   แล้วอาจ "พลาด" จังหวะที่ผู้ทดสอบเดินผ่าน ทำให้เวลา checkpoint เพี้ยน

   ปกติช่วงนี้กินเวลาแค่ไม่กี่วินาที (เดิน 3 เมตร) จึงไม่ถึงเกณฑ์ 45 วินาที
   แต่ถ้าการทดสอบล้มเหลว (ไม่มีใครเดินผ่านเลย) ช่วงนี้อาจยาวถึง 130 วินาที
   และเว็บจะเห็นเป็น OFFLINE

   วิธีแสดงผลที่แนะนำ: ถ้า device_status/chair.state เป็น RUNNING อยู่
   ให้แสดง Checkpoint เป็น "กำลังทดสอบ" แทนคำว่า OFFLINE
   และดูค่า chair.checkpoint_online ประกอบ — ค่านั้นมาจาก ESP-NOW โดยตรง
   จึงยังแม่นยำอยู่แม้ Checkpoint จะหยุดส่ง heartbeat

⚠️ pending_uploads > 0 หมายถึงอะไร
   แปลว่าบอร์ดทดสอบเสร็จแล้วแต่อัปโหลดไม่ได้ (เน็ตมีปัญหา) ผลถูกเก็บไว้ใน
   หน่วยความจำถาวรของ ESP32 และจะส่งเองอัตโนมัติเมื่อกลับมาออนไลน์
   เว็บควรขึ้นแบนเนอร์เตือน เพราะแปลว่ารายการที่เห็นบนหน้าจอ "ยังไม่ครบ"
   ⚠️ บอร์ดเก็บได้สูงสุด 8 รายการ ถ้าเกินจะทิ้งรายการเก่าสุด — ถ้าเห็นเลข 8
      ควรเตือนแรงกว่าปกติว่าอาจมีข้อมูลสูญหายแล้ว

[5] Collection: device_commands  (เว็บเขียน / บอร์ดอ่านแล้วเขียน ack กลับ)
================================================================================

มี 2 document: "chair" และ "checkpoint"

รูปแบบคำสั่งทั้งหมดใช้แพตเทิร์นเดียวกัน คือคู่ของ
    <cmd>_requested_at   (เว็บเขียน — epoch วินาที)
    <cmd>_handled_at     (บอร์ดเขียนกลับ — epoch วินาที)
บอร์ดจะทำคำสั่งก็ต่อเมื่อ requested_at > handled_at

  ทำไมต้องใช้แพตเทิร์นนี้ ไม่ใช้ boolean ธรรมดา
  เพราะบอร์ดรีบูตได้ (RAM หายหมด) Firestore เป็น state เดียวที่รอด
  ถ้าใช้ flag แบบ pending=true บอร์ดจะไม่มีทางรู้ว่า "เคยทำคำสั่งนี้ไปแล้วหรือยัง"
  หลังบูตขึ้นมา — โดยเฉพาะคำสั่ง reset ที่จะกลายเป็นวนรีบูตไม่รู้จบ
  การเทียบ timestamp ทำให้ทุกคำสั่งมีเอกลักษณ์และ idempotent

device_commands/chair — fields ที่เว็บเขียนได้:

  subject_key             stringValue    รหัสผู้เข้าทดสอบคนปัจจุบัน
  session_id              stringValue    รหัส session ปัจจุบัน
  reset_requested_at      integerValue   สั่งรีบูตบอร์ด
  start_requested_at      integerValue   สั่งเริ่มการทดสอบ
  abort_requested_at      integerValue   สั่งยกเลิกการทดสอบที่กำลังทำอยู่

  fields ที่บอร์ดเขียนกลับ (เว็บอ่านได้ ห้ามเขียนทับ):
  reset_handled_at, start_handled_at, abort_handled_at

device_commands/checkpoint:
  reset_requested_at / reset_handled_at   (รองรับแค่ reset)

--------------------------------------------------------------------------------
5.1 คำสั่ง START — ความหมายไม่ใช่ "เริ่มจับเวลาเดี๋ยวนี้"
--------------------------------------------------------------------------------
START = "อนุญาตให้เริ่มจับเวลาได้" (arm) ตัวจับเวลาจริงจะเริ่มตอนที่เซนเซอร์
ตรวจพบว่าผู้ทดสอบลุกขึ้นจากเก้าอี้จริง ๆ

เหตุผล: เอกสารหัวข้อ 5.4.2 ระบุว่าให้ "จับเวลาตั้งแต่เริ่มเห็นผู้สูงอายุขยับตัว
หลังจากได้ยินสัญญาณเริ่ม" ถ้าจับเวลาตอนกดปุ่มบนเว็บ จะรวมเวลาที่ผู้ทดสอบยัง
นั่งฟังคำสั่งอยู่เข้าไปด้วย ทำให้ค่าที่ได้ยาวกว่าความจริงและเทียบกับเกณฑ์
มาตรฐานไม่ได้

⚠️ ค่าเริ่มต้นของเฟิร์มแวร์ตอนนี้คือ REQUIRE_WEB_START = 0
   แปลว่าบอร์ด "เริ่มเองอัตโนมัติ" เมื่อมีคนลุก โดยไม่ต้องรอเว็บกด Start
   ปุ่ม Start บนเว็บจะยังส่งคำสั่งได้ปกติแต่ไม่มีผลอะไร (บอร์ด armed อยู่แล้ว)

   ถ้าต้องการให้ปุ่ม Start บังคับใช้จริง (เจ้าหน้าที่ต้องกดก่อนทุกรอบ
   ตามเอกสารขั้นตอนที่ 3) ให้แก้ใน ESP_Chair_v2.ino เป็น
       #define REQUIRE_WEB_START  1
   แล้วอัปโหลดใหม่ — ดูค่าปัจจุบันได้จาก field `armed` ใน device_status/chair

--------------------------------------------------------------------------------
5.2 คำสั่ง ABORT
--------------------------------------------------------------------------------
ใช้เมื่อผู้ทดสอบล้ม / ขอหยุด / ทดสอบผิดพลาด
บอร์ดจะหยุดจับเวลา สั่ง Checkpoint ให้เคลียร์ไฟ และบันทึกผลเป็น status="aborted"

⚠️ ข้อจำกัดเรื่องเวลาตอบสนอง (สำคัญ — ต้องออกแบบ UI เผื่อไว้)
   บอร์ดจะ "ไม่" เช็คคำสั่งเลยในช่วง STATE_RETURNING (ช่วงที่ผู้ทดสอบเดินกลับ
   มาหาเก้าอี้) เพราะการเรียก HTTPS บล็อกลูปหลายร้อยมิลลิวินาที ซึ่งจะทำให้
   การตรวจจับ "นั่งลง" ช้าไปด้วย และเวลาที่วัดได้จะคลาดเคลื่อน — ยอมไม่ได้
   สำหรับเครื่องมือวัดทางการแพทย์

   สรุปเวลาตอบสนองของ Abort:
     • ช่วง READY / RUNNING (เดินออก) : ประมาณ 0-4 วินาที ✅
     • ช่วง RETURNING (เดินกลับ)      : จะทำงานหลังผู้ทดสอบนั่งลงแล้ว
                                        หรือหลังหมดเวลา 120 วินาที
   UI ควรแสดงว่า "กำลังส่งคำสั่งยกเลิก..." แล้วรอ abort_handled_at เปลี่ยนค่า
   ไม่ใช่ขึ้นว่า "ยกเลิกแล้ว" ทันทีที่กดปุ่ม

   หมายเหตุ: ถ้าผู้ทดสอบล้มจริง เขาจะไม่กลับมานั่ง ระบบจะ timeout เองใน 120
   วินาทีและบันทึกเป็น aborted อยู่ดี ไม่ค้างถาวรเหมือนเวอร์ชันเดิม

--------------------------------------------------------------------------------
5.3 คำสั่ง RESET
--------------------------------------------------------------------------------
สั่งรีบูตบอร์ด ใช้ตอนบอร์ดมีปัญหา
บอร์ดจะเขียน reset_handled_at ให้สำเร็จก่อน แล้วค่อยรีบูต ถ้าเขียนไม่สำเร็จ
จะไม่รีบูตและลองใหม่รอบหน้า (กันวนรีบูตไม่รู้จบ)

หลังรีบูต บอร์ดใช้เวลาประมาณ 15-25 วินาทีจึงจะกลับมาออนไลน์
(WiFi + NTP + Firebase token) UI ควรแสดง countdown ไม่ใช่ขึ้น error ทันที

⚠️ ต้องรีเซ็ตทั้งสองบอร์ด ให้เขียนคำสั่งลงทั้ง device_commands/chair
   และ device_commands/checkpoint — ไม่งั้นบอร์ดที่ไม่ได้รีเซ็ตจะยังจำ
   สถานะเดิมอยู่ (แต่ Chair จะส่ง ABORT ไปให้ Checkpoint ก่อนรีบูตอยู่แล้ว
   จึงไม่ค้างในกรณีปกติ)

--------------------------------------------------------------------------------
5.4 ตัวอย่างโค้ดฝั่งเว็บ
--------------------------------------------------------------------------------
  import { doc, setDoc, getDoc } from "firebase/firestore";

  const nowEpoch = () => Math.floor(Date.now() / 1000);

  // ⚠️ ใช้ setDoc + merge:true เสมอ ห้ามใช้ setDoc เปล่า ๆ
  //    เพราะ setDoc เปล่าจะลบ field *_handled_at ที่บอร์ดเขียนไว้ทิ้งหมด
  //    ทำให้บอร์ดเห็นคำสั่งเก่าเป็น "ยังไม่ได้ทำ" แล้วทำซ้ำ
  //    (เคสร้ายแรงสุดคือ reset — จะรีบูตซ้ำโดยไม่มีใครสั่ง)
  async function sendCommand(device, cmd) {
    await setDoc(
      doc(db, "device_commands", device),
      { [`${cmd}_requested_at`]: nowEpoch() },
      { merge: true }
    );
  }

  // เริ่ม session ใหม่ (ทดสอบ 3 รอบ)
  async function startSession(subjectKey) {
    const sessionId = `${subjectKey}-${nowEpoch()}`;
    await setDoc(
      doc(db, "device_commands", "chair"),
      { subject_key: subjectKey, session_id: sessionId },
      { merge: true }
    );
    return sessionId;
  }

[6] ⚠️ สิ่งที่เว็บ "ต้องทำใหม่" — ลงทะเบียนผู้เข้าทดสอบ
================================================================================

เอกสารโครงการขั้นตอนที่ 1 กำหนดให้ลงทะเบียนผู้เข้าทดสอบก่อน (ชื่อ อายุ ส่วนสูง
เพศ ประวัติสุขภาพ อุปกรณ์ช่วยเดิน) และผูกกับรหัสประจำตัว (PID)

ปัจจุบันเฟิร์มแวร์ยังส่งผลออกมาโดยไม่รู้ว่าเป็นของใคร (subject_key =
"unassigned") เพราะไม่มีใครบอกมัน — **เว็บต้องเป็นคนบอก**

สิ่งที่เว็บต้องทำ:
  1. ทำหน้าลงทะเบียน/เลือกผู้เข้าทดสอบ เก็บใน collection ของเว็บเอง
     (แนะนำ: subjects/{subject_key})
  2. ก่อนเริ่มทดสอบ เขียน subject_key + session_id ลง device_commands/chair
  3. บอร์ดจะแนบสองค่านี้ไปกับผลทุกรอบโดยอัตโนมัติ

⚠️ ข้อกำหนด PDPA (เอกสารหัวข้อ "มาตรการด้านความปลอดภัย")
   subject_key ต้องเป็น **รหัสสุ่ม** ห้ามใช้ชื่อ-นามสกุล เลขบัตรประชาชน
   หรือเบอร์โทร เพราะค่านี้จะถูกส่งไปเก็บบนบอร์ดและอยู่ใน document ผลการทดสอบ
   เอกสารระบุให้ใช้ Pseudonymization และ "ไม่เก็บชื่อ-นามสกุลในคลังวิจัย"
   ชื่อจริงให้เก็บแยกในตาราง subjects ที่จำกัดสิทธิ์เข้าถึงต่างหาก

   ตัวอย่างที่ดี : "SUBJ-7f3a91"
   ตัวอย่างที่ผิด: "somchai-jaidee", "1103700123456"

⚠️ ความยาวสูงสุด 31 ตัวอักษร (ทั้ง subject_key และ session_id)
   บอร์ดเก็บใน buffer ขนาดคงที่ ถ้าเกินจะถูกตัดท้ายทิ้ง ทำให้ผลผูกผิดคนได้


================================================================================
[7] ⚠️ สิ่งที่เว็บ "ต้องทำใหม่" — ทดสอบ 3 รอบแล้วเฉลี่ย
================================================================================

เอกสารหน้าแรกระบุ: "โดยเราจะทำการทดสอบ 3 ครั้ง แล้วนำผลของเวลาในการทดสอบมาเฉลี่ย"

บอร์ดส่งผลมาทีละรอบ (แต่ละรอบเป็น 1 document) พร้อม session_id และ trial_no
**การจับกลุ่มและเฉลี่ยต้องทำที่เว็บ** เฟิร์มแวร์ไม่ได้คำนวณให้

ตรรกะที่เว็บต้องมี:
  1. query tug_results where session_id == X and status == "completed"
  2. ถ้าได้ครบ 3 รอบ → เฉลี่ย total_sec → คำนวณ risk_level จากค่าเฉลี่ย
     (ใช้เกณฑ์ 11 / 30 เดียวกัน)
  3. แสดง "ผลสรุปของ session" จากค่าเฉลี่ย และแสดงผลรายรอบเป็นรายละเอียดย่อย
  4. ถ้ามีรอบ aborted ให้แสดงว่ารอบนั้นล้มเหลว และต้องทำเพิ่มให้ครบ 3 รอบ
     ที่ completed

⚠️ อย่าเอา risk_level ของแต่ละรอบมาโหวตกัน ให้เฉลี่ยเวลาก่อนแล้วค่อยตัดเกณฑ์
   ผลจะไม่ตรงกับวิธีของเอกสาร

⚠️ บอร์ดนับ trial_no ให้เองและรีเซ็ตกลับเป็น 1 เมื่อ session_id เปลี่ยน
   ถ้าเว็บไม่เปลี่ยน session_id เลย เลข trial จะวิ่งขึ้นเรื่อย ๆ ไม่หยุด
   → เว็บต้องสร้าง session_id ใหม่ทุกครั้งที่เริ่มทดสอบคนใหม่หรือชุดใหม่


================================================================================
[8] ⚠️ กับดักเรื่องชนิดข้อมูล — พลาดกันบ่อยมาก
================================================================================

8.1 integerValue ของ Firestore REST เก็บเป็น "string"
    บอร์ดเขียนผ่าน REST API ค่าตัวเลขจำนวนเต็มทั้งหมดจึงถูกเก็บเป็น
    integerValue ซึ่งใน JSON คือ string เช่น "1753012345"

    Firebase JS SDK จะแปลงกลับมาให้เป็น number อัตโนมัติในกรณีปกติ
    แต่ถ้าเว็บอ่านผ่าน REST API ตรง ๆ หรือ Cloud Function ต้อง parseInt() เอง
    → ถ้าเจอ bug แบบ "เวลาเป็น NaN" หรือ "เปรียบเทียบแล้วผลแปลก" ให้สงสัยจุดนี้ก่อน

8.2 หน่วยเวลาเป็น "วินาที" ไม่ใช่ milliseconds
    started_at / finished_at / last_seen / *_requested_at ทั้งหมดเป็น epoch วินาที
    JavaScript ใช้ milliseconds → ต้องคูณ 1000 ก่อนสร้าง Date object

        new Date(finished_at * 1000)      ✅ ถูก
        new Date(finished_at)             ❌ ผิด (จะได้ปี 1970)

    ส่วน checkpoint_sec / return_sec / total_sec เป็น doubleValue หน่วยวินาที
    ใช้ได้ตรง ๆ ไม่ต้องแปลง

8.3 เวลาบนบอร์ดเป็น UTC
    บอร์ดเรียก configTime(0, 0, ...) คือ offset 0 = UTC
    เว็บต้องแปลงเป็นเวลาไทย (UTC+7) ตอนแสดงผลเอง
    ถ้าไม่แปลง ผลการทดสอบตอนบ่าย 3 จะขึ้นเป็น 8 โมงเช้า


================================================================================
[9] Firestore Security Rules — ต้องแก้
================================================================================

ปัจจุบันบอร์ดล็อกอินด้วย Email/Password ของ Firebase Auth หนึ่งบัญชี
(บัญชีอุปกรณ์ ไม่ใช่บัญชีผู้ใช้) เว็บต้องตั้ง rules ให้:

  • บัญชีอุปกรณ์: เขียน tug_results ได้, เขียน device_status ได้,
    อ่าน device_commands ได้, เขียนได้เฉพาะ field *_handled_at
  • บัญชีเจ้าหน้าที่: อ่าน tug_results / device_status ได้,
    เขียน device_commands ได้ (เฉพาะ *_requested_at, subject_key, session_id)
  • ห้ามเปิด public read บน tug_results เด็ดขาด — เป็นข้อมูลสุขภาพ
    เอกสารระบุให้ทำตาม PDPA และ HIPAA

⚠️ ปัญหาที่ต้องคุยกันในทีม
   รหัสผ่านบัญชีอุปกรณ์ถูก compile ฝังอยู่ในเฟิร์มแวร์ (โดยตั้งใจ — ซอร์สโค้ด
   ฝั่ง Arduino ไม่ได้ขึ้น git และมี dev คนเดียวเป็นคนอัปโหลด) แต่ใครที่เข้าถึง
   ตัวบอร์ดได้ก็ยังถอด flash ออกมาอ่านรหัสได้อยู่ดี

   ดังนั้น rules ต้องตั้งบนสมมติฐานว่า "บัญชีอุปกรณ์อาจถูกขโมย" คือ
   จำกัดสิทธิ์ให้เขียนได้เฉพาะ path ของตัวเองเท่านั้น และห้ามลบ/แก้ document
   ผลการทดสอบที่เขียนไปแล้ว
   ระยะยาวควรแยกบัญชีต่อบอร์ด (chair กับ checkpoint คนละบัญชี) เพื่อให้
   บอร์ดหนึ่งที่ถูกเจาะ ทำอะไรกับข้อมูลของอีกบอร์ดไม่ได้

[10] Index ที่ต้องสร้าง
================================================================================

query ที่เว็บจะใช้บ่อยและต้องมี composite index:

  1. tug_results: session_id (ASC) + trial_no (ASC)
     → ดึงผลทั้ง 3 รอบของ session หนึ่ง เรียงตามลำดับรอบ

  2. tug_results: subject_key (ASC) + finished_at (DESC)
     → ดูประวัติย้อนหลังของคนหนึ่ง ทำกราฟแนวโน้ม (เอกสารขั้นตอนที่ 6)

  3. tug_results: status (ASC) + finished_at (DESC)
     → หน้ารายการผลล่าสุด กรองเอาเฉพาะ completed

Firebase จะขึ้น error พร้อมลิงก์สร้าง index ให้อัตโนมัติตอนรัน query ครั้งแรก
กดลิงก์นั้นได้เลย

[12] สิ่งที่ยังไม่มีในเฟิร์มแวร์ — ถ้าเว็บต้องการ ต้องแจ้งก่อน
================================================================================

เอกสารโครงการกำหนดไว้แต่ฮาร์ดแวร์ชุดปัจจุบันยังทำไม่ได้:

  • split times ครบ 5 ช่วง (sit-to-stand / walk-out / turn / return / sit-down)
    ตอนนี้ได้แค่ 3 ค่า: checkpoint_sec, return_sec, total_sec
    การแยก sit-to-stand กับ turn ต้องใช้ load-cell หรือกล้อง pose estimation
    ตามที่เอกสารออกแบบไว้ ซึ่งยังไม่ได้ติดตั้ง

  • ตัวชี้วัดการเดิน (cadence, step variability, sway, turn duration)
    ต้องใช้ IMU หรือ computer vision — ยังไม่มีในต้นแบบนี้

  • ข้อควรรู้สำหรับการเขียนรายงาน: เอกสารออกแบบให้เก้าอี้ใช้ load-cell
    (เซนเซอร์วัดแรงกด) แต่ต้นแบบจริงใช้ ultrasonic วัดระยะแทน
    เกณฑ์ของเอกสารที่ว่า "หยุดเวลาเมื่อผู้สูงอายุลงนั่งหลังพิงพนักเรียบร้อย"
    จึงวัดตรง ๆ ไม่ได้ ระบบใช้ "มีวัตถุอยู่ภายใน 10 ซม. ต่อเนื่อง 500ms" แทน
    ควรระบุเป็นข้อจำกัดของงานไว้ในรายงาน

  • ระบบยังไม่รองรับหลายเก้าอี้/หลายห้องพร้อมกัน
    device_status และ device_commands ใช้ document ID ตายตัวว่า "chair"
    ถ้าจะขยายเป็นหลายชุด ต้องเปลี่ยนเป็น device_status/{deviceId}
    และเฟิร์มแวร์ต้องมี DEVICE_ID ของตัวเอง — แจ้งฝั่ง Arduino ก่อน
